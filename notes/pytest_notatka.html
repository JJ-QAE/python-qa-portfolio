Podstawy
schemat funkcja + test
Złota zasada: funkcja w kodzie, test osobno
Funkcja przyjmuje parametry i zwraca wynik. Test sprawdza czy wynik się zgadza. Nigdy nie używaj input() ani print() w funkcji którą testujesz!
# kod.py — funkcja bez input() i print()
def bmi_kalkulator(waga, wzrost):
    return waga / (wzrost * wzrost)

# test_kod.py — test sprawdza wynik
def test_bmi():
    assert round(bmi_kalkulator(70, 1.75), 2) == 22.86

# Zasada: funkcja zwraca wynik → test sprawdza wynik z assert
# round() przy liczbach z przecinkiem — inaczej 22.86 != 22.857142...
uruchomienie
Uruchom testy
W terminalu — uruchamia wszystkie testy w pliku.
# jeden plik:
pytest test_kod.py

# wszystkie testy w folderze:
pytest

# wynik:
# PASSED ✅ — test przeszedł
# FAILED ❌ — test nie przeszedł
assert
Sprawdź wynik
assert sprawdza czy coś jest prawdą. Przy liczbach z przecinkiem zawsze używaj round()!
# liczby całkowite:
assert dodaj(2, 3) == 5

# liczby z przecinkiem → round()!
assert round(bmi_kalkulator(70, 1.75), 2) == 22.86

# tekst:
assert sprawdz_bmi(17.5) == "niedowaga"

# True/False:
assert sprawdz_wiek(17) == False

parametrize
Wiele przypadków, jeden test
Zamiast pisać 3 osobne testy z różnymi danymi — piszesz jeden. Dekorator musi być przyklejony bezpośrednio nad def. Nazwy w stringu i w def muszą być identyczne. Zasada: kopiujesz test i zmieniasz tylko liczby? → użyj parametrize.
import pytest

# BEZ parametrize — powtarzasz kod (źle)
def test_bmi_1(): assert round(bmi_kalkulator(70, 1.75), 2) == 22.86
def test_bmi_2(): assert round(bmi_kalkulator(50, 1.60), 2) == 19.53

# Z parametrize — jeden test, trzy uruchomienia (dobrze)
@pytest.mark.parametrize("waga, wzrost, oczekiwane", [
    (70, 1.75, 22.86),   # → uruchomienie 1
    (50, 1.60, 19.53),   # → uruchomienie 2
    (90, 1.80, 27.78)    # → uruchomienie 3
])
def test_bmi_parametrize(waga, wzrost, oczekiwane):
    assert round(bmi_kalkulator(waga, wzrost), 2) == oczekiwane

# Uwaga: oczekiwane wartości musisz policzyć ręcznie!
# pytest nie wie czy twoje liczby są poprawne
@pytest.fixture
fixture
Dane testowe do reużycia
Fixture przygotowuje dane raz — używasz w wielu testach. Nazwa fixture musi być identyczna z parametrem funkcji testowej. Literówka w nazwie = pytest nie znajdzie danych!
import pytest

# 1. Definiujesz fixture — dane w jednym miejscu
@pytest.fixture
def przykladowy_pacjent():
    return {"waga": 80, "wzrost": 1.67}

# 2. Używasz fixture jako parametr testu (ta sama nazwa!)
def test_bmi_normalny(przykladowy_pacjent):
    assert round(bmi_kalkulator(
        przykladowy_pacjent["waga"],
        przykladowy_pacjent["wzrost"]
    ), 2) == 28.69

# fixture = dane startowe dla wielu testów
# parametrize = jeden test, wiele zestawów danych
pytest.raises()
raises
Testuj błędy
Sprawdza czy funkcja poprawnie się wysypuje przy złych danych. Jeśli błąd wystąpi — PASSED. Jeśli nie wystąpi — FAILED.
import pytest

# wzrost=0 → ZeroDivisionError
def test_zero_wzrost():
    with pytest.raises(ZeroDivisionError):
        bmi_kalkulator(70, 0)

# jeśli błąd wystąpi → PASSED ✅
# jeśli błąd NIE wystąpi → FAILED ❌
conftest.py i pytest.ini
conftest.py
Fixtures dla wielu plików
Fixture w conftest.py jest dostępna dla WSZYSTKICH plików testowych w folderze — bez importowania. Nazwa musi być dokładnie "conftest.py" — literówka i pytest jej nie znajdzie!
# conftest.py — fixtures w jednym miejscu
import pytest
import requests

@pytest.fixture
def api_post():
    return requests.get("https://jsonplaceholder.typicode.com/posts/1")

# api_test6.py — używa fixture bez importowania!
def test_status(api_post):
    assert api_post.status_code == 200

# Struktura projektu:
# ClaudeLessons/
# ├── conftest.py     ← fixtures dla wszystkich
# ├── api_test4.py    ← może używać fixtures z conftest
# └── api_test6.py    ← może używać fixtures z conftest
pytest.ini
Konfiguracja pytest
pytest domyślnie szuka plików test_*.py. Jeśli twoje pliki nazywają się inaczej (np. api_test*.py), musisz dodać pytest.ini. Sprawdź co widzi: pytest --co -q
# pytest.ini — w głównym folderze projektu
[pytest]
python_files = test_*.py api_test*.py
Fixture Scopes
scope
Kiedy fixture ma się uruchamiać
Scope mówi pytest kiedy fixture ma się uruchomić. Dla testów API zawsze używaj scope="session" — mniej zapytań do serwera, szybsze testy.
import pytest
import requests

# function (domyślny) — uruchamia się przed KAŻDYM testem
@pytest.fixture(scope="function")
def api_function():
    print(">>> odpalam przed każdym testem")
    return requests.get("https://jsonplaceholder.typicode.com/posts/1")

# session — uruchamia się RAZ dla całej sesji testów
@pytest.fixture(scope="session")
def api_session():
    print(">>> odpalam tylko raz!")
    return requests.get("https://jsonplaceholder.typicode.com/posts/1")

# scope="function" → 10 testów = 10 zapytań do API
# scope="session"  → 10 testów = 1 zapytanie do API
Scope Kiedy się uruchamia Kiedy używać
function przed każdym testem dane które mogą się zmieniać
session raz dla wszystkich testów zapytania API, wolne operacje
Markers
@pytest.mark
Oznaczaj i filtruj testy
Markers pozwalają oznaczać testy i uruchamiać je wybiórczo. Własne markery musisz zarejestrować w pytest.ini — inaczej pytest pokaże warning.
import pytest

# skip — pomiń test z powodem
@pytest.mark.skip(reason="API tymczasowo niedostępne")
def test_pominiety():
    assert response.status_code == 200

# slow — własny marker dla wolnych testów
@pytest.mark.slow
def test_wolny():
    response = requests.get("https://jsonplaceholder.typicode.com/posts")
    assert len(response.json()) == 100

# uruchamianie z markerami:
# pytest -m slow          → tylko testy slow
# pytest -m "not slow"    → wszystkie OPRÓCZ slow
# pytest -v -s            → szczegółowy wynik + printy
pytest.ini
Rejestracja markerów
Bez rejestracji pytest pokaże warning o nieznanym markerze.

Monkeypatch
monkeypatch
Testuj bez internetu — podmień prawdziwe na fałszywe
Monkeypatch podmienia prawdziwe zapytanie API na fałszywe — test działa zawsze, bez internetu. Używaj gdy API jest niestabilne, płatne, lub testy mają działać w CI/CD.
import pytest
import requests

# 1. Fałszywa odpowiedź — udaje prawdziwy response
class FakeResponse:
    status_code = 200
    def json(self):
        return {"id": 1, "title": "Fake Post", "userId": 1}

# 2. monkeypatch podmienia requests.get na funkcję zwracającą FakeResponse
def test_get_post(monkeypatch):
    def fake_get(url):
        return FakeResponse()

    monkeypatch.setattr(requests, "get", fake_get)

    response = requests.get("https://jsonplaceholder.typicode.com/posts/1")
    assert response.status_code == 200
    assert response.json()["title"] == "Fake Post"

# bez monkeypatch → 0.6s+ (prawdziwe zapytanie do API)
# z monkeypatch   → 0.02s  (brak zapytania — zawsze działa!)

# Uwaga: test jest POZA klasą FakeResponse!
Złota zasada pytest: funkcja przyjmuje parametry i zwraca wynik — nigdy input() ani print() w środku. Test sprawdza wynik z assert. Przy liczbach z przecinkiem zawsze round(). Oczekiwane wartości w parametrize musisz policzyć ręcznie. scope="session" = fixture odpala się raz dla całej sesji. monkeypatch = podmień prawdziwe na fałszywe.
