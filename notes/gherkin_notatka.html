Behavior-Driven Development – testy pisane ludzkim językiem, uruchamiane przez Pythona

Co to jest BDD i Gherkin?

BDD (Behavior-Driven Development) to podejście do testowania, w którym scenariusze testowe są pisane w języku naturalnym — zrozumiałym dla programistów, testerów i product ownerów.

Gherkin to język używany do pisania tych scenariuszy. Pliki mają rozszerzenie .feature. Sam w sobie to tylko tekst — żeby działał jako test, potrzebuje frameworka.

Przykład pliku .feature

Feature: Logowanie użytkownika

  # Background działa jak setup – wykonuje się przed każdym scenariuszem
  Background:
    Given użytkownik jest na stronie logowania

  Scenario: Pomyślne logowanie
    When wpisuje login "jowita@test.com" i hasło "Admin123"
    Then zostaje przekierowany na stronę główną
    And widzi komunikat "Witaj, Jowita!"

  Scenario: Błędne hasło
    When wpisuje login "jowita@test.com" i hasło "złehasło"
    Then widzi błąd "Nieprawidłowe dane logowania"

  # Scenario Outline = parametryzacja (jak @pytest.mark.parametrize)
  Scenario Outline: Walidacja formularza
    When wpisuje login "<login>" i hasło "<hasło>"
    Then widzi błąd "<komunikat>"

    Examples:
      | login           | hasło    | komunikat                  |
      |                 | Admin123 | Login nie może być pusty   |
      | jowita@test.com |          | Hasło nie może być puste   |
      | niemail         | Admin123 | Podaj prawidłowy email     |
⚠️ Tekst w pliku .feature musi być identyczny z tekstem w dekoratorach Pythona — co do literki. Inaczej pytest-bdd rzuci błędem.

Struktura projektu

projekt/ ├── features/ │ └── logowanie.feature ← plik z scenariuszami ├── tests/ │ └── test_logowanie.py ← implementacja kroków ├── conftest.py ← wspólne fixtures └── pytest.ini ← konfiguracja pytest

Implementacja kroków w Pythonie

conftest.py — wspólny fixture

import pytest

@pytest.fixture
def dane_logowania():
    # Słownik przekazywany między krokami Given → When → Then
    return {}

test_logowanie.py — kroki

from pytest_bdd import scenarios, given, when, then, parsers

# Podpinamy plik .feature
scenarios("../features/logowanie.feature")

# Symulowana baza użytkowników
UŻYTKOWNICY = {
    "jowita@test.com": "Admin123"
}

# ── GIVEN ──────────────────────────────────────────────────────
@given(parsers.parse('użytkownik podaje login "{login}" i hasło "{hasło}"'))
def podaj_dane(dane_logowania, login, hasło):
    dane_logowania["login"] = login
    dane_logowania["hasło"] = hasło

# ── WHEN ───────────────────────────────────────────────────────
@when("system sprawdza dane")
def sprawdz_dane():
    pass  # logika jest w then

# ── THEN ───────────────────────────────────────────────────────
@then("użytkownik zostaje zalogowany")
def zalogowany(dane_logowania):
    login = dane_logowania["login"]
    hasło = dane_logowania["hasło"]
    assert UŻYTKOWNICY.get(login) == hasło

@then(parsers.parse('użytkownik widzi błąd "{komunikat}"'))
def blad_logowania(dane_logowania, komunikat):
    login = dane_logowania["login"]
    hasło = dane_logowania["hasło"]
    assert UŻYTKOWNICY.get(login) != hasło
💡 parsers.parse(...) służy do wyciągania zmiennych z tekstu. Cudzysłowy "{login}" w pliku .feature odpowiadają parametrom funkcji Pythona o tej samej nazwie.
