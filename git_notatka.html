Jak to działa
Git vs GitHub
Różnica
Git to narzędzie na twoim komputerze które śledzi zmiany w kodzie. GitHub to strona internetowa gdzie przechowujesz kod w chmurze i pokazujesz go światu.
# Git  → program na twoim komputerze (lokalnie)
# GitHub → strona w internecie (zdalnie)

# Twój komputer                GitHub
# ClaudeLessons/    ──push──▶  JJ-QAE/python-qa-portfolio
# api_test4.py      ◀─pull──   api_test4.py
Trzy strefy
Jak Git śledzi pliki
Git ma trzy miejsca gdzie może być twój kod. Musisz przejść przez wszystkie trzy żeby plik trafił na GitHub.
# 1. Working Directory — pliki na dysku (jeszcze nieśledzone)
# 2. Staging Area      — pliki dodane przez git add (gotowe do commit)
# 3. Repository        — pliki zapisane przez git commit (historia)

# Droga pliku na GitHub:
# dysk ──git add──▶ staging ──git commit──▶ repo ──git push──▶ GitHub
Podstawowe komendy
git add
Dodaj do staging
Mówisz Gitowi które pliki chcesz zapisać. Bez tego commit nic nie zapisze.
git add api_test4.py        # jeden plik
git add plik1.py plik2.py   # kilka plików
git add .                   # WSZYSTKIE zmiany

# sprawdź co jest w staging:
git status
git commit
Zapisz zmiany
Zapisuje snapshot kodu z opisem. Wiadomość powinna opisywać CO zrobiłaś, nie jak.
git commit -m "opis zmian"

# dobre wiadomości:
# "pytest BMI - parametrize i raises"
# "testy API - POST, status 201"
# "conftest.py - shared fixture"

# złe wiadomości:
# "zmiany" / "update" / "fix"
git push
Wyślij na GitHub
Wysyła wszystkie commity na GitHub. Bez push GitHub nie widzi twoich zmian.
git push                    # wyślij na GitHub

# pełna sekwencja zawsze:
git add plik.py
git commit -m "opis"
git push
git pull
Pobierz z GitHub
Pobiera zmiany z GitHub na twój komputer. Używasz gdy pracujesz na kilku komputerach lub w zespole.
git pull                    # pobierz zmiany

# zawsze rób pull przed push
# gdy pracujesz w zespole!
Sprawdzanie stanu
git status
Co się zmieniło?
Pokazuje które pliki są zmodyfikowane, które w staging, które nieśledzone. Uruchamiaj często!
git status

# możliwe statusy pliku:
# "Changes to be committed"    → w staging (git add zrobiony)
# "Changes not staged"         → zmieniony, ale bez git add
# "Untracked files"            → nowy plik, Git go nie śledzi
git log
Historia commitów
Pokazuje historię wszystkich zmian w projekcie.
git log                     # pełna historia
git log --oneline           # skrócona historia

# przykład:
# ebb0f63 conftest.py - shared fixture
# 186cde0 testy API - POST, status 201
# 63e84ce pytest fixture - przykladowy_pacjent BMI
Usuwanie i cofanie
git rm
Usuń plik z GitHuba
Usuwa plik z GitHuba i z dysku. Pamiętaj żeby potem zrobić commit i push!
git rm nazwa_pliku.py
git commit -m "usuwam błędny plik"
git push

# jeśli chcesz usunąć tylko z GitHuba
# ale zostawić na dysku:
git rm --cached nazwa_pliku.py
git restore
Cofnij zmiany
Cofa niezapisane zmiany w pliku — przywraca do ostatniego commita. Uwaga: nie da się tego odwrócić!
git restore plik.py         # cofnij zmiany w pliku
git restore --staged plik.py # cofnij git add
Konfiguracja projektu
pytest.ini
Mówisz pytest co szukać
pytest domyślnie szuka tylko plików test_*.py i *_test.py. Żeby znalazł też api_test*.py potrzebny jest pytest.ini w głównym folderze projektu.
# pytest.ini — w folderze ClaudeLessons/
[pytest]
python_files = test_*.py api_test*.py

# sprawdź czy pytest widzi wszystkie pliki:
pytest --co -q
.gitignore
Czego nie wysyłać na GitHub
Plik .gitignore mówi Gitowi żeby ignorował niektóre pliki — np. cache, klucze API, foldery IDE. Ważne dla bezpieczeństwa!
# .gitignore — w folderze ClaudeLessons/
__pycache__/
.pytest_cache/
.idea/
*.pyc
.env                        # ← tutaj trzymaj klucze API!

# NIGDY nie wrzucaj kluczy API na GitHub!
# Jeśli przypadkowo wrzuciłaś — zmień klucz natychmiast

Złota zasada Git: add → commit → push. Zawsze w tej kolejności. Commit bez push = GitHub tego nie widzi. Push bez commit = nic się nie zmieni.
